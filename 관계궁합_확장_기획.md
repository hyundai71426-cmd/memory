# 관계별 궁합 모드 확장 + 궁합 랭킹 기획

기존 4.6(궁합 모드)을 "두 사람" 단일형에서 **관계 유형별 콘텐츠**로 확장하고, 4.7(오늘의 랭킹)과 같은 구조의 랭킹을 관계유형별로 추가하는 설계.

## 1. 관계 유형 (1차 4종)

| 유형 | 사용 가중치 (`타로카드_가중치.md`) | 톤 포인트 |
|---|---|---|
| 연인궁합 | 애정 | 설레는/달콤한 톤 |
| 베프궁합 | 애정 (우정 전용 가중치가 없어 애정 컬럼 재사용) | 친근하고 가벼운 톤, "연애" 표현 대신 "케미/우정" 표현으로 치환 |
| 상사궁합 | 직장 | 격식 있는 톤, 평가하듯 들리는 표현 지양("이 분과는 같이 일하기 좋은 흐름이에요" 식) |
| 짝사랑궁합 | 애정 | 조심스럽고 희망적인 톤. **단정적 결론 금지** — "지금은 ~한 흐름" 정도로, 결과가 관계 자체를 단정하지 않도록 주의 (안전장치 4번 참고) |

- 4가지 모두 산출 로직은 4.6과 동일(시드: 두 사람 생년월일 정렬 + 오늘 날짜 → 베이스 점수 + 케미 카드의 해당 가중치 컬럼). 유형마다 **사용하는 가중치 컬럼과 해석 톤**만 달라짐.
- `/compatibility` 페이지에 진입 시 관계 유형 선택 탭/버튼을 먼저 노출 → 선택 후 생년월일 2개 입력.

## 2. 궁합 랭킹 — 관계유형별 TOP5

- 4.7(오늘의 랭킹)과 동일한 구조를 관계유형마다 따로 운영: **연인궁합 TOP5 / 베프궁합 TOP5 / 상사궁합 TOP5 / 짝사랑궁합 TOP5** — 총 4개의 독립 리더보드.
- 진입 조건: 해당 유형의 오늘자 궁합 점수가 그 유형 TOP5 안에 들면 "랭킹 등록하기" 노출.
- 등록 시 **두 사람 각각 영문 3자 이니셜**을 입력해 `KSM ♥ ABC` 형태로 표시(비속어 필터는 4.7과 동일하게 양쪽 모두에 적용).
- 동점자 처리·자정 초기화 정책은 4.7과 동일(선착순 우선).
- DB 구조: 4.7에서 이미 도입하기로 한 서버 DB에 `relation_type` 컬럼만 추가해 테이블 재사용(연인/베프/상사/짝사랑/개인종합 5종을 한 테이블에서 구분) — 새 인프라가 추가로 필요하진 않음.

## 3. 흐름 정리

1. `/compatibility` 진입 → 관계 유형 선택(연인/베프/상사/짝사랑).
2. 두 사람 생년월일 입력(정렬 처리는 기존과 동일, 순서 무관).
3. 베이스 점수 + 케미 카드(해당 유형 가중치 컬럼) → 최종 궁합 점수 산출, 해석 텍스트 노출.
4. 그 유형의 오늘자 TOP5 안에 들면 이니셜 등록 유도 → 랭킹 즉시 반영.
5. 결과 화면에 "다른 관계로도 보기"(예: 연인 본 다음 베프궁합도) 버튼 제공 — 체류시간 증대(`체류시간_증대_기획.md`)와 결합되는 자연스러운 다음 동작.

## 4. 안전장치 (짝사랑궁합 특이사항)

- 짝사랑궁합은 관계가 아직 성립되지 않은 민감한 주제이므로, 결과 텍스트에서 상대의 마음을 단정하는 표현(예: "그 사람은 당신을 안 좋아해요") 절대 금지. "지금 흐름은 ~", "조금 더 다가가 보면 좋을 시기" 같은 가능성 중심 문구로 한정.
- 운세텍스트_생성방식.md의 금칙어 필터·안내 문구 정책을 그대로 적용하고, 짝사랑궁합 전용 프롬프트 가드레일 한 줄을 추가로 둠.

## 5. 기획안 반영 필요 사항

- 4.6(궁합 모드)에 관계 유형 4종 및 가중치 매핑 표 추가, 본 문서 링크.
- 4.7(오늘의 랭킹) 또는 신설 서브섹션에 "관계유형별 궁합 랭킹"이 같은 DB를 `relation_type` 컬럼으로 확장해 재사용한다는 점 명시.
- `/compatibility` 페이지 설명에 관계 유형 선택 단계 추가.
